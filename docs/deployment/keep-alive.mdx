Free tier hosting services can sleep when idle. memory404 uses GitHub Actions to keep HTTP services warm and health routes to verify MongoDB connectivity.

## Problem

| Service | Sleep threshold | Impact |
|---------|----------------|--------|
| Render (free) | ~15 min idle | 30–60s cold start on first request |
| Vercel | Stays warm | Cold starts possible without pings |
| Supabase Auth | Provider-managed | Health ping verifies Auth availability |

## GitHub Actions workflow

The workflow at `.github/workflows/keep-alive.yml` pings services on a schedule:

| Service | Schedule | Endpoint |
|---------|----------|----------|
| Render | Every 14 minutes | `{RENDER_API_URL}/health` |
| Vercel | Every 14 minutes | `{VERCEL_APP_URL}/api/health` |
| Supabase Auth | Daily | `{SUPABASE_PROJECT_URL}/auth/v1/health` |

### Setup

Set these as GitHub repository variables:

| Variable | Example |
|----------|---------|
| `VERCEL_APP_URL` | `https://memory404.vercel.app` |
| `RENDER_API_URL` | `https://memory404-api.onrender.com` |
| `SUPABASE_PROJECT_URL` | `https://<project-ref>.supabase.co` |

### Manual trigger

Go to **Actions → Keep services alive → Run workflow** in GitHub.

## Database health

The Vercel and Render health routes issue a MongoDB `ping` through the native driver. Therefore, scheduled HTTP checks also validate that `MONGODB_URI` is configured and Atlas is reachable. MongoDB Atlas is persistence; Supabase remains Auth-only.

## Render upgrade

For production reliability, upgrade Render to the **Starter** plan ($7/mo) for always-on with no spin-down.

## Monitoring

Verify keep-alive is working:

```bash
# Check Render uptime in health response
curl https://memory404-api.onrender.com/health

# Response includes uptime in seconds
{ "ok": true, "uptime": 86400.5 }
```

If uptime resets frequently, the service is spinning down between pings.
