memory404 uses Supabase only for authentication and session refresh. Persistent application data lives in MongoDB Atlas through the native MongoDB driver; Supabase Postgres is not used by the running application.

## Create a project

1. Go to [supabase.com](https://supabase.com) and create a new project
2. Open **Project Settings → API**
3. Copy the project URL and publishable key

## Environment variables

```bash
NEXT_PUBLIC_SUPABASE_URL=https://<project-ref>.supabase.co
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_DEFAULT_KEY=<publishable-key>
```

Set these in `.env.local` for development and in Vercel for production. The publishable key is intended for client use, but it still should be managed through environment configuration. Never expose service-role keys.

`NEXT_PUBLIC_SUPABASE_ANON_KEY` remains supported as a legacy fallback.

## Current auth behavior

The middleware in `utils/supabase/` refreshes sessions. API code resolves a signed-in Supabase user and mirrors the user's ID and email into MongoDB's `users` collection so `groups` and `links` can be scoped by `userId`.

Authentication is not yet enforced for every route: when no session exists, the application currently uses a development-user fallback. Do not treat this fallback as production access control.

## Persistence

Configure persistence separately:

- `MONGODB_URI` — secret MongoDB Atlas connection string
- `MONGODB_DB` — optional database name, default `memory404`

Do not configure `DATABASE_URL` or `DIRECT_URL` for normal development or deployment.

See [Database schema](/database/schema) for the MongoDB collection model and [Environment variables](/deployment/environment-variables) for the full configuration reference.
